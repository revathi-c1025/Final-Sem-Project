SLIDE 2 — INTRODUCTION
────────────────────────────────────────────────────────────────────────────────
Title: Introduction — The Problem We Are Solving

NOTES:
"Let me start with the problem.

Dell OME has thousands of manual test cases documented in qTest Manager.
Each test case describes what needs to be tested — but there are NO executable
scripts. A test engineer has to manually read each test case, understand the
OME framework APIs, and hand-write a Python script. This takes days per test.

On top of that:
- When a test fails, engineers manually debug logs and re-run.
- The atlas-ome-staging framework changes frequently, breaking old scripts.
- There is no systematic way to classify failures or learn from past fixes.

Our solution addresses all of this:
- Automatically fetches steps from qTest via REST API.
- Generates valid pytest code using AI or deterministic templates.
- Executes on a real OME appliance.
- Classifies failures into 8 error categories and auto-fixes.
- Retries up to 3 times with improved code each time.

This transforms a process that took days into one that takes seconds."

+ atlas-ome-staging framework integration + LLM/RAG-based generation +
agentic auto-fix in a single, real-world enterprise pipeline.

That is the exact gap this project fills."

SLIDE 4 — SYSTEM ARCHITECTURE OVERVIEW
────────────────────────────────────────────────────────────────────────────────
Title: System Architecture Overview — Five Cooperating Agents

NOTES:
"The system is built as five cooperating agents, each with a single
well-defined responsibility, coordinated by a master OrchestratorAgent.

1. QTestAgent: Reads test case data from qTest Manager via REST API.
   Calls GET /api/v3/projects/{id}/test-cases/{id}/test-steps.
   Returns structured JSON with step descriptions and expected results.

2. TestGeneratorAgent: Takes the steps, classifies them using regex-based
   keyword matching, optionally retrieves RAG context, and generates a
   complete OmeBaseFunctionalTestCase pytest file.

3. TestExecutorAgent: Runs the generated file via pytest subprocess,
   capturing stdout, stderr, and log files per attempt.

4. FixerAgent: When a test fails, classifies the error into one of 8
   categories (import, syntax, SSL, connection, auth, assertion, timeout,
   attribute) and produces a ranked list of fix strategies.

5. OrchestratorAgent: The master — coordinates the full pipeline, manages
   the retry loop (up to MAX_RETRIES = 3), and produces the final JSON + HTML
   report.

The pipeline is:
FETCH → GENERATE → EXECUTE → [FAIL? → ANALYZE → REGENERATE → RETRY] → REPORT

External integrations:
- qTest Manager (cloud) on the left
- atlas-ome-staging framework + OME Appliance on the right
- Optional LLM (GPT-4/Azure) for AI-powered generation"

SLIDE 6 — AI / ML & RAG COMPONENTS
────────────────────────────────────────────────────────────────────────────────
Title: AI / ML & RAG Components — How Intelligence Is Built In

NOTES:
"There are three AI/ML layers in the system.

LLM Layer:
When an API key is available, we use GPT-4 (OpenAI or Azure endpoint).
The model receives: the TC steps, error context from previous failures,
and RAG-retrieved code examples.
Temperature is set to 0.2 for deterministic, reproducible code output.
If no LLM is configured, the system falls back to a deterministic
template engine — ensuring the system works fully offline.

RAG System:
The Retrieval-Augmented Generation component uses:
- Sentence-Transformers: all-MiniLM-L6-v2 model (384-dimensional embeddings)
- FAISS: IndexFlatL2 for nearest-neighbour vector search
- Knowledge Base: atlas-ome test patterns chunked at 500 tokens with 50-token overlap
- Retrieval: Top-3 most relevant chunks injected into the LLM prompt

Memory:
The system maintains several forms of memory:
- Knowledge base files (persistent on disk)
- FAISS index (persisted for reuse)
- Per-agent event logs (in-memory, exported to JSON report)
- Fix history (FixerAgent tracks all past failure analyses)

DEMO NOTE: In the demo, FAISS was temporarily disabled due to a Windows
paging file memory limitation. The template engine took over and all tests
still passed. This is a known issue with a planned fix."

SLIDE 7 — TEST GENERATION PIPELINE
────────────────────────────────────────────────────────────────────────────────
Title: Test Generation Pipeline — qTest Steps → Executable pytest Module

NOTES:
"Let me walk through exactly how a test case becomes executable code.

Step 1 — FETCH:
QTestAgent calls /api/v3/projects/334/test-cases/{id}/test-steps.
Returns JSON with 'description' and 'expected' for each step.

Step 2 — CLASSIFY:
Each step description is regex-matched against a classifier that maps
keywords to atlas-ome-staging framework methods.
Examples:
  'install plugin'   → PluginServiceTask.plugins_actions()
  'discover device'  → DiscoveryConfig.add_devices()
  'create group'     → GroupServiceTask.create_and_verify_static_group()
  'register SCOM'    → ConsoleServiceTask.add_console_montblanc()

Step 3 — RAG LOOKUP:
Relevant test code examples retrieved from FAISS knowledge base.
Provides concrete implementation patterns to the LLM.

Step 4 — GENERATE:
LLM or template produces a complete OmeBaseFunctionalTestCase class.
The class has three lifecycle methods:
  pre_testcase() — prerequisites and setup
  run_test()     — step-by-step implementation with LOGGER.tc_step()
  post_testcase() — cleanup (delete groups, disconnect consoles)

Step 5 — WRITE FILE:
Saved as test_TC_{ID}.py — STABLE filename, overwrites on each retry.
No timestamps — same file is regenerated, not duplicated.

The code example (bottom left) shows the actual structure generated
for TC-174995."

SLIDE 8 — AUTO-FIX ENGINE
────────────────────────────────────────────────────────────────────────────────
Title: Auto-Fix Engine — FixerAgent: Intelligent Failure Classification

NOTES:
"When a test fails, the FixerAgent takes over. Let me explain the retry loop
and error classification.

RETRY LOOP (left panel):
1. Execute test via pytest
2. If PASS → done, report success
3. If FAIL → FixerAgent classifies the error
4. TestGeneratorAgent regenerates with error context
5. Retry from step 1 (up to MAX_RETRIES = 3)
6. If still failing after 3 attempts → mark FAILED, record details

ERROR CLASSIFICATION TABLE:
The FixerAgent uses regex pattern matching on the error traceback and
stderr to classify failures into one of 8 categories:

- import_error (0.90 confidence): Missing module → add import statement
- syntax_error (0.90): Python syntax problem → fix in regenerated code
- ssl_error (0.90): SSL cert verification → add verify=False
- connection_error (0.80): Network unreachable → add retry + timeout
- auth_error (0.70): 401/403 → fix auth headers
- assertion_error (0.70): Wrong assertion logic → fix or soften
- timeout_error (0.80): Too slow → increase timeout to 600s
- attribute_error (0.80): Wrong key/attr access → fix access pattern
- unknown_error (0.50): Full regeneration fallback

The coloured dots in the table show: green = high confidence fix,
orange = medium, red = best-effort fallback.

REAL EXAMPLE: TC-180015 had a parent_id mismatch (400 Bad Request).
Classified as assertion_error → fix strategy generated → correct
parent_id=1021 added → test PASSED."

SLIDE 11 — CHALLENGES & HOW WE FIXED THEM
────────────────────────────────────────────────────────────────────────────────
Title: Challenges Faced & How We Fixed Them

NOTES:
"Every research project faces unexpected challenges. Let me walk through the
six major ones we encountered and how each was resolved.

1. Atlas Framework Import Failure [FIXED]
   The atlas_framework package is an internal Dell package that cannot be
   installed via pip. Every generated test was failing with ModuleNotFoundError.
   FIX: Created OmeApiClient — a standalone REST wrapper that replicates the
   framework interface. Tests now use _FRAMEWORK_AVAILABLE guard and skip
   gracefully when atlas_ome is not installed.

2. RAG System OOM on Windows [WORKAROUND]
   Loading the SentenceTransformer model + FAISS index exhausted the Windows
   paging file. Error: 'paging file too small for this operation to complete'.
   FIX: Temporarily disabled RAG initialization in app.py. Template engine
   handles all generation. Planned: switch to a lighter model (e.g., paraphrase-MiniLM-L3-v2).

3. Web UI Module Caching [WORKAROUND]
   Flask was caching the old shopease_framework import even after we replaced
   the code with SimpleTestGenerator. Multiple cache clearing attempts failed.
   FIX: Used CLI mode for demo. Planned: subprocess-based generation with fresh
   Python process per request.

4. Static Group parent_id = Wrong [FIXED]
   TC-180015 was failing with HTTP 400 on group creation. Traced the OME API
   response and found the required parent_id = 1021, not the default.
   FIX: Hardcoded correct parent_id; added a fallback to query current IDs.

5. Edit Repo API No-Op [FIXED]
   The PATCH firmware catalog endpoint returned HTTP 200 but did not rename
   the repository. This was an OME version-specific behaviour.
   FIX: Added fallback — verify rename took effect; if not, keep original name.
   Logged as known OME API limitation.

6. Template Brace Escaping [FIXED]
   Python's .format() method interprets {variable} in strings. The generated
   code body strings also contained {variable} patterns for framework calls.
   FIX: _BODY_* constants use single braces; only the outer template string
   uses double braces {{ }} for literal brace output."

PREPARED Q&A ANSWERS:
Q: Why not use existing test automation tools like Selenium/Appium?
A: OME is a server management system, not a web app. It uses proprietary
   REST APIs and the atlas-ome-staging framework. Generic tools don't understand
   OME concepts like device discovery, plugin management, or firmware catalogs.

Q: How does the RAG system handle new framework versions?
A: The knowledge base is rebuilt by re-indexing the atlas-ome-staging codebase.
   FAISS index is regenerated. No model retraining needed.

Q: What happens if the LLM generates incorrect code?
A: The FixerAgent classifies the failure, provides error context, and
   triggers regeneration with corrected instructions. Up to 3 retries.

Q: How scalable is the system?
A: The OrchestratorAgent's run_batch() method processes multiple TCs
   sequentially. Parallelism can be added with concurrent.futures.

Q: What is the estimated time saving vs manual scripting?
A: A manual script for TC-174995 (11 steps) would take ~1-2 days.
   The automated system generates and executes in under 2 minutes.
   This is a 100-200x speedup in script creation time.

Q: Why choose Python / pytest over other frameworks?
A: atlas-ome-staging is Python-based. pytest is the standard at Dell for
   OME testing. All existing reference tests use pytest patterns.

Q: Is the qTest token secure?
A: For demo, token is in config.py. For production, it would be in
   environment variables or a secrets manager (e.g., Azure Key Vault).
